PROJECT 1: SOC Threat Detection Log Analyzer

30-Second Elevator Pitch

"I built a Python-based SOC Threat Detection Log Analyzer that acts as a lightweight SIEM. It ingests Linux authentication logs, parses them using custom Regex patterns, and runs them through a detection engine. It uses a sliding-time window to detect brute force attacks, flags suspicious cross-geolocation logins, and identifies abnormal after-hours access, outputting actionable incident reports in JSON format."

Architecture Diagram

[Linux /var/log/auth.log] ↓ (LogParser: Regex matching, Timestamp extraction) ↓ (DetectionEngine: Memory-based sliding window & sets) ├── Rule 1: Brute Force (>10 fails in 5m) ├── Rule 2: Suspicious IP (New IP for known user) └── Rule 3: Abnormal Time (Outside 8AM-6PM) ↓ (AlertManager: Console formatting & JSON export) ↓ [incident_report.json]

Component Breakdown

  • main.py: The entry point. Uses argparse to provide a CLI interface. Orchestrates the pipeline: Read -> Parse -> Detect -> Alert.
  • log_parser.py: Contains specific Regex patterns to match syslog formats. Dynamically injects the current year to handle syslog's missing year issue in datetimes. Extracts Timestamp, User, IP, and Status.
  • detection_engine.py: The brains.
    - Brute Force: Tracks login attempts using collections.deque and timedelta to implement a 5-minute sliding window.
    - Suspicious Login: Maintains a set() mapping known Users to IPs to flag anomalies.
    - Time Check: Uses datetime.hour to flag non-business hour access.
  • alert_manager.py: Handles the presentation layer, printing color-coded terminal output and serializing dictionaries to incident_report.json for downstream API consumption.

Design Decisions

DecisionWhy? / Rationale
Native Regex Parser Kept the application lightweight with zero external dependencies. Faster than heavy ML parsers for structured auth logs.
Sliding Time Window Using a strict 5-minute window rather than total counts prevents false positives. 10 fails over a month is normal; 10 fails in 5 minutes is an attack.
JSON Output Machine-readable format. In a real cloud environment, this JSON could be easily shipped to AWS S3 or ingested by an AWS Lambda function for further orchestration.
Q: How does this project map to Cloud Engineering?
"In a cloud environment, you deal with massive amounts of CloudTrail or VPC Flow logs. Building this script taught me how logging pipelines work fundamentally—ingestion, parsing, and alerting. I can apply this same Python logic inside an AWS Lambda function triggered by an S3 event."
Q: Why did you use Python instead of Bash?
"Bash is great for simple grepping, but Python's `datetime` and `timedelta` modules were required for the complex 5-minute sliding window implementation. Python also allows easy scaling to APIs and JSON serialization."
Q: How did you handle massive log files?
"Instead of loading the entire file into memory with readlines(), I processed the file line-by-line using a generator setup, ensuring the memory footprint stays low even if the log file is gigabytes in size."

PROJECT 2: SOAR-Lite (Security Orchestration, Automation & Response)

30-Second Elevator Pitch

"I built an end-to-end SOAR platform. It ingests security events into an SQLite database, evaluates them against a JSON-configurable rule engine, and triggers automated remediation like blocking malicious IPs. It features a Flask web dashboard for SOC analysts to monitor alerts, view blocked IPs, and export forensics data to CSV, demonstrating my ability to build automated, full-stack security tools."

Architecture Diagram

[Security Event Logs] ↓ (LogCollector.py: Parses JSON/Syslog events) ↓ [SQLite Database: 4 Relational Tables] ↓ (Engine.py: Evaluates JSON Rules → Alert Trigger) ↓ (Automation: Executes BLOCK_IP, Inserts to DB) ↓ [Flask Dashboard (app.py): Metrics, Alerts, CSV Export]

Component Breakdown

  • database.py: Implements sqlite3. Uses init_db() to create a relational schema with 4 tables: logs, alerts, blocked_ips, and response_actions.
  • collector.py: Reads mock incoming event types (LoginFailure, LoginSuccess, PortScan, Access) and normalizes them for the database.
  • engine.py: Unlike Project 1, rules are loaded from an external JSON file, meaning rules can be updated without touching code. When a rule matches, it automatically executes a response action (simulated firewall block) and writes it to the DB.
  • app.py: A Flask application routing 6 endpoints (`/`, `/logs`, `/alerts`, etc.). Uses HTML/Jinja2 templates for the UI and pandas for generating CSV exports for analyst forensics.

Database Schema

Table NameColumns / Purpose
logs id, timestamp, source_ip, event_type, details, severity (Raw ingestion)
alerts id, log_id, rule_name, alert_time, status (Flags raised by engine)
blocked_ips id, ip_address, block_time, reason, unblock_time (Active remediations)
response_actions id, alert_id, action_type, execute_time, status (Audit history of SOAR actions)

Q: What is the "Orchestration" part of your SOAR project?
"The orchestration connects the detection layer to the response layer automatically. Instead of an analyst reading an alert and manually updating a firewall rule, the engine orchestrates that workflow independently based on the JSON configs."
Q: Why did you use SQLite and not Postgres/MySQL?
"For a lightweight, portable proof-of-concept, SQLite was perfect because it requires no separate server setup and stores state in a single file. However, in a production AWS environment, I would migrate this to Amazon RDS (PostgreSQL) or DynamoDB for high availability and concurrent writes."
Q: How would you deploy this to AWS?
"I would containerize the Flask app with Docker, deploy it to ECS (Elastic Container Service) or EKS, use a managed RDS instance for the database, and place an Application Load Balancer in front of it to handle TLS termination."
